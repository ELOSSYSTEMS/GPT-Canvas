



Does running the bootloader really do anything

Interrogate the term first.

"Running the bootloader" is not a single uniform action. It refers to executing the code that initializes hardware and loads the operating system (OS) or firmware. It is not optional; it is structurally required in any digital system where persistent storage and runtime execution are separated. Now refine:

I. Clarify Context
Are you referring to:

Embedded systems (e.g., microcontrollers)?

Mobile devices (e.g., Android bootloader unlocking)?

General-purpose computers (e.g., GRUB on Linux)?

Custom firmware environments (e.g., U-Boot, Coreboot)?

"Running" in each case has different implications.

II. In Bare-Metal Terms
A bootloader does the following (unless stripped by design):

Stage 1: Initializes CPU state, memory controllers, basic peripherals.

Stage 2: Loads and verifies the OS kernel or a next-stage loader.

Optionally: Performs secure boot validation, cryptographic checks, or config loading.

Without this sequence:

The system has no instruction pointer redirection.

No OS, kernel, or runtime environment is ever loaded.

You have silicon without software — inert.

So yes, bootloader execution is not optional. The question is what flavor of action you are referring to.

III. If You Mean "Running the Bootloader Manually"
In systems where the bootloader can be manually invoked (e.g., Android fastboot, BIOS boot menu), the outcome varies:

It can allow:

Flashing firmware.

Unlocking security settings.

Loading recovery or alternate OS.

But "running" in that sense is not doing what it usually does at power-on. It's running maintenance code.

In that case, the purpose shifts from booting the OS to configuring boot behavior.

Bottom Line
Running the bootloader is always doing something — usually everything.
The only case where it seems like "nothing happens" is when it silently and successfully loads the next stage. You only question its role when something breaks.
